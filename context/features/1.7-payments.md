# Feature 1.7 — Payments

**Branch:** `feature/1.7-payments`
**Roadmap ref:** Phase 1 · Feature 1.7 — Database migrations — payments
**Status:** 🟢 Complete

## Overview

Introduces the payments layer: escrow accounts that hold brand funds per campaign, payout records tracking every transfer to creators, and a single-row `platform_settings` config table the admin panel manages. Also closes the deferred `entry_ripple_earnings.payout_id` FK introduced in 1.6 now that `payouts` exists.

## Migrations

| File | Purpose |
|---|---|
| `2026_04_15_120000_create_escrow_transactions_table.php` | Per-campaign escrow record with held/released/refunded totals, Stripe PaymentIntent id, status state machine |
| `2026_04_15_120001_create_payouts_table.php` | One row per transfer to a creator — gross/fee/net, payout_type enum, Stripe transfer id, retry tracking |
| `2026_04_15_120002_create_platform_settings_table.php` | Single-row config (`id = 1`), default 15% fee + 50/25/15/10 contest splits |
| `2026_04_15_120003_add_payout_id_fk_to_entry_ripple_earnings_table.php` | Closes deferred FK from 1.6 |

## Models

- `EscrowTransaction` — belongsTo Campaign, BrandProfile. Decimal casts on amount columns, datetime casts on held_at / fully_released_at.
- `Payout` — belongsTo Entry, CreatorProfile, Campaign. HasMany EntryRippleEarning (reverse of 1.6). Decimal casts on gross/fee/net.
- `PlatformSetting` — `$incrementing = false`, `CREATED_AT = null`. Static `PlatformSetting::current()` returns the single row (firstOrCreate with defaults) so callers don't need to know the id.

## Seeder

`PlatformSettingSeeder` inserts the default config row (15% fee, 50/25/15/10 split, $25 min payout, $10 referral bonuses). Wired into `DatabaseSeeder`.

## Snapshot / fee model

- `campaigns.platform_fee_pct` (1.5) is the per-campaign snapshot of `platform_settings.platform_fee_pct` at publish time — admin changes to the platform rate never retroactively affect live campaigns.
- Payouts store `gross_amount`, `platform_fee`, and `net_amount` explicitly so the split is auditable without recomputation.

## Inverse relations added

- `Campaign::escrowTransaction()` (HasOne), `Campaign::payouts()` (HasMany)
- `CreatorProfile::payouts()`, `Entry::payouts()`
- `EntryRippleEarning::payout()` (BelongsTo, now that payout_id FK exists)
