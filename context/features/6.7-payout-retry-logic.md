# Feature 6.7 — Payout Retry Logic

**Phase:** 6 — Payouts  
**Status:** Complete  
**Branch:** feature/1.13-1.14-onboarding

---

## Overview

Failed payouts are automatically retried once after a 30-minute delay. If a second failure occurs, the payout is permanently marked failed and all admin users receive an email alert.

---

## Design

Laravel's job `$tries` is not used here because `PayoutService::executeTransfer()` catches all Stripe exceptions internally — the job itself never throws. Retry is managed manually inside `markFailed()`.

---

## Flow

```
executeTransfer() → Stripe error caught
  → markFailed(payout, reason)
      retry_count < 2 (1st failure):
        payout.failure_reason updated, retry_count = 1
        pending_earnings untouched (payout stays pending)
        ProcessPayoutJob::dispatch(payoutId)->delay(30 min)
      retry_count >= 2 (2nd failure):
        payout.status = failed
        pending_earnings decremented
        Log::error(...)
        User::role('admin')->each → PayoutFailureAlert notification sent
```

---

## Changes

### `app/Services/PayoutService.php` — `markFailed()`

- 1st failure: keep `pending` status, record `failure_reason` + `retry_count`, reschedule job with 30-min delay
- 2nd failure: mark `failed`, decrement `pending_earnings`, log, notify all admin users

### `app/Notifications/PayoutFailureAlert.php` (new)

- Mail notification sent to each admin user
- Includes: payout ID, creator name/email, net amount, payout type, failure reason
- Links to `/admin/payouts` for review
