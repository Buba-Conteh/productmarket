## Overview

Schema for the campaign layer. A base `campaigns` table holds the shared fields (brief, status, deadline, fee snapshot) and three type-specific detail tables split Contest / Ripple / Pitch — avoiding a wide table full of nullable columns. Adds pivots for allowed platforms and content types, the co-branded sponsorship table, and the Pitch-only application table.

## What shipped

### Migrations (8)
| Table | Purpose |
|---|---|
| `campaigns` | Base table — brand FK, type enum (contest/ripple/pitch), title, brief, JSON requirements/hashtags/regions/links, `platform_fee_pct` snapshot, status enum (draft → pending_escrow → active → closed → completed / cancelled), deadline, `max_creators`, `ai_brief_used` flag. Indexed on `(status, type)` and `deadline` |
| `campaign_platforms` | Composite-PK pivot — campaigns ↔ platforms |
| `campaign_content_types` | Composite-PK pivot — campaigns ↔ content_types |
| `campaign_contest_details` | 1:1 per Contest campaign — prize_amount, runner_up_prize, winner_entry_id (FK added in 1.6), winner_selected_at, selection_notes. No timestamps |
| `campaign_ripple_details` | 1:1 per Ripple campaign — initial_fee, rpm_rate, milestone_interval, max_payout_per_creator, total_budget, budget_spent. No timestamps |
| `campaign_pitch_details` | 1:1 per Pitch campaign — product name/description/url, product_images JSON, budget_cap, min_bid, max_bid. No timestamps |
| `campaign_co_brands` | Multiple co-sponsors per campaign — contribution_amount, contribution_pct, status enum (invited/accepted/declined). Unique on (campaign_id, brand_profile_id) |
| `campaign_applications` | Pitch campaign applications before entry submission — pitch text, status enum (pending/approved/rejected). Unique on (campaign_id, creator_profile_id) |

### Models
- `App\Models\Campaign` — base model with relations: `brand()`, `platforms()`, `contentTypes()`, `contestDetails()`, `rippleDetails()`, `pitchDetails()`, `coBrands()`, `applications()`. JSON columns cast to `array`; `platform_fee_pct` as decimal:2
- `App\Models\CampaignContestDetail`, `CampaignRippleDetail`, `CampaignPitchDetail` — type-specific details, all `public $timestamps = false` per schema
- `App\Models\CampaignCoBrand`, `CampaignApplication`
- `BrandProfile` → added `campaigns()` hasMany relation
- `CreatorProfile` → added `campaignApplications()` hasMany relation

## Design decisions

- **Three detail tables instead of nullable columns** — per schema spec; keeps base table clean, queries fast, and each detail table can evolve independently.
- **`platform_fee_pct` snapshot on `campaigns`** — fee rate is captured at campaign creation so future admin changes to `platform_settings.platform_fee_pct` never retroactively affect live campaigns.
- **`winner_entry_id` not yet a FK** — `entries` table doesn't exist until feature 1.6. Stored as raw nullable `ulid` for now; the FK constraint can be added in a follow-up migration during 1.6 if desired.
- **Detail tables skip timestamps** — 1:1 with parent `campaigns`; the campaign's timestamps cover them.
- **Indexes on `(status, type)` and `deadline`** — the two filter combinations that hit the discovery feed + cron jobs hardest.
- **Pivots use composite PKs** — no synthetic id needed; prevents duplicate rows by construction.
- **JSON fields only for fixed-shape value lists** (hashtags, region codes, URLs). Relations that need queryable semantics (platforms, content types) are pivots.
