# Feature 6.2 — Platform Fee Deduction Logic

**Phase:** 6 — Payouts  
**Status:** Complete  
**Branch:** feature/1.13-1.14-onboarding

---

## Overview

Pure backend service layer that all payout features (6.3–6.6) depend on. Handles the math (gross → fee → net), creates `Payout` records, and executes Stripe Connect transfers — keeping that logic in one place so no payout type re-implements it.

---

## Implementation

### `app/DTOs/PayoutAmounts.php`

Immutable value object with four fields: `gross`, `fee`, `net` (decimal strings), and `feePct`. Provides `netInCents()` and `grossInCents()` helpers for Stripe API calls.

### `app/Services/PayoutService.php`

Three public methods:

#### `calculateAmounts(string $gross, string $feePct): PayoutAmounts`

Pure math using `bcmath` (no floating-point drift). Formula:

```
fee = gross × feePct / 100
net = gross − fee
```

#### `createPayout(Entry $entry, string $payoutType, string $gross): Payout`

- Reads `platform_fee_pct` from `$entry->campaign` — the **snapshotted** rate set at campaign creation (rate changes never affect live campaigns)
- Calls `calculateAmounts()`
- Creates a `Payout` record with `status = pending`
- Increments `creator_profiles.pending_earnings` by `net` inside a DB transaction

#### `executeTransfer(Payout $payout): void`

- Guards: creator must have an `active` Stripe Connect account
- Calls `stripe->transfers->create()` with `net_amount` in cents, `destination = stripe_connect_id`, `transfer_group = campaign_id`
- **On success** (DB transaction):
  - `payout.status = paid`, stores `stripe_transfer_id` and `paid_at`
  - `creator.pending_earnings -= net`, `creator.total_earned += net`
  - `escrow_transaction.total_released += gross`
- **On failure** (catches `ApiErrorException`):
  - Logs the error
  - `payout.status = failed`, stores `failure_reason`, increments `retry_count`
  - Reverts `creator.pending_earnings -= net`

---

## Design Decisions

- **Fee snapshot**: `campaign.platform_fee_pct` is used, not `PlatformSetting::current()`. The schema specifies rate changes never touch live campaigns.
- **bcmath for arithmetic**: avoids floating-point precision errors on monetary values.
- **`pending_earnings` lifecycle**: incremented on `createPayout`, moved to `total_earned` on success, reverted on failure. Retry logic (6.7) is responsible for re-incrementing on re-try.
- **Does not throw on transfer failure**: callers check `payout->status` after `executeTransfer` returns; 6.7 handles retry/alerting.

---

## Usage (by future payout features)

```php
// e.g. in Contest prize payout (6.3):
$payout = $payoutService->createPayout($entry, 'contest_prize', '1000.00');
$payoutService->executeTransfer($payout);
```
