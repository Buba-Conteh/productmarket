## Overview

Schema for creator submissions. One `entries` base table covers all three campaign types (contest/ripple/pitch), with per-platform posting data split out into a pivot that also tracks verified view counts. Pitch-specific bid data, the Ripple earnings audit trail, edit-request back-and-forth, and the raw API sync log all live in their own tables. Also closes the deferred FK on `campaign_contest_details.winner_entry_id` now that `entries` exists.

## What shipped

### Migrations (7)
| Table | Purpose |
|---|---|
| `entries` | Base table — campaign/creator/content_type FKs, type enum, video_url, duration, caption, tags JSON, requirements_acknowledged, status enum (draft → pending_review → approved/rejected → live → won/not_selected/disqualified), timestamps. Unique(campaign_id, creator_profile_id) — one entry per creator per campaign |
| `entry_platforms` | Composite-PK pivot tracking per-platform posting: posted_url, verified_view_count (bigint), last_synced_at |
| `entry_pitch_details` | 1:1 with Pitch entries — proposed_bid, accepted_bid, pitch text, bid_accepted_at. No timestamps |
| `entry_ripple_earnings` | Audit trail of every Ripple earning event — milestone_number, views_at_milestone, amount, type enum (initial_fee/milestone), triggered_at, payout_id (FK deferred to feature 1.7). No timestamps |
| `entry_edit_requests` | Brand revision requests — requested_by_user_id, notes, status enum (pending/addressed/dismissed), addressed_at |
| `view_sync_logs` | Raw audit for every API sync job run — view_count, synced_at, success bool, error_message. No timestamps |
| *migration 7* | Adds the foreign key on `campaign_contest_details.winner_entry_id` → `entries.id` (null on entry delete) |

### Models
- `App\Models\Entry` — relations to campaign, creator, contentType, platforms (with pivot columns), pitchDetails, rippleEarnings, editRequests, viewSyncLogs. Casts: `tags` → array, `requirements_acknowledged` → bool, all three timestamp fields → datetime
- `App\Models\EntryPitchDetail`, `EntryRippleEarning`, `EntryEditRequest`, `ViewSyncLog` — all `final`, `HasUlids`, with `timestamps = false` where schema specifies
- `Campaign` → added `entries()` hasMany
- `CreatorProfile` → added `entries()` hasMany
- `CampaignContestDetail` → added `winnerEntry()` belongsTo

## Design decisions

- **Unique(campaign_id, creator_profile_id)** on `entries` — enforces one entry per creator per campaign at the DB level.
- **Per-platform view counts live on the pivot**, not the entry — a creator cross-posts to multiple platforms and each has its own running count and sync timestamp.
- **`verified_view_count` is bigint** — accommodates viral videos past 2B views that would overflow a signed int.
- **`entry_ripple_earnings.payout_id` is a bare ulid for now** — `payouts` table arrives in feature 1.7, FK will be added then (same pattern used for winner_entry_id between 1.5 → 1.6).
- **Separate `view_sync_logs` table** — every API hit logged even on failure, so ops can diagnose token expiry / rate limits / platform outages without touching the hot `entry_platforms` row.
- **No timestamps on earnings/sync_logs/detail tables** — their purpose-specific time columns (`triggered_at`, `synced_at`, `bid_accepted_at`) are sufficient; Laravel's auto-timestamps would be redundant.
- **Indexes on `entries(status, type)`, `entry_ripple_earnings(entry_id, milestone_number)`, `view_sync_logs(entry_id, synced_at)`** — the three hot-path queries (dashboard list filters, milestone dedup check, per-entry sync history).
